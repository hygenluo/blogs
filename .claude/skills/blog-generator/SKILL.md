# 博客生成 Skill

此 skill 帮助用户在 Mizuki 博客项目中创建高质量、风格一致的博客文章。

## 工作流程总览

```
用户提供内容/需求
      │
      ▼
┌─────────────────┐
│ 1. 识别博客类型   │ ← 判断属于哪种博客（hot100/题解/Agent/教程/随笔...）
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ 2. 确定 Frontmatter│ ← 生成 tags、category、description 等元数据
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ 3. 确定存放位置   │ ← 选择合适的文件夹（需用户确认）
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ 4. 编写博客内容   │ ← 按对应模板和风格要求撰写
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ 5. 用户审核      │ ← 展示完整博客，等用户确认
└─────────────────┘
      │
      ▼
┌─────────────────┐
│ 6. 发布 (Git)    │ ← 用户确认后，git add → commit → push
└─────────────────┘
```

## 第一阶段：识别博客类型

根据用户提供的信息判断博客类型，这决定了后续所有步骤：

| 用户提供的内容 | 博客类型 | 详细模板 |
|:-:|:-:|:-:|
| LeetCode 链接含 `top-100-liked` | **Hot100 题解** | 模板 1 |
| LeetCode 链接不含 hot100 / 面试题 / 算法题 | **Algorithm and Structure 题解** | 模板 2 |
| Agent、LangGraph、RAG 等 AI 框架代码/资料 | **Agent 技术教程** | 模板 3 |
| Python 基础语法/编程入门内容 | **PyLearning 教程** | 模板 4 |
| NLP、Transformer、深度学习理论 | **NLP 知识博客** | 模板 5 |
| 工具使用教程 (FastAPI/Git 等) | **Guides 教程** | 模板 6 |
| 读书笔记/学习笔记 | **Learning 笔记** | 模板 7 |
| 个人日记 | **日记** | 模板 7 |
| 随笔/杂谈 | **随笔** | 模板 7 |

**不确定时**：如果用户没有提供足够信息来判定类型，主动询问。比如"你想写什么类型的博客？是算法题解、技术教程还是随笔？"

详细模板结构参考 `references/blog-templates.md`。

## 第二阶段：确定 Frontmatter

### 必填字段（每篇博客都需要）

```yaml
title: <标题>
author: Hygen
published: <今天日期 YYYY-MM-DD>
description: <一句话描述>
```

### 标题 (title) 规则

- **Hot100 / 题解**：直接使用题目名称，如"两数之和""三数之和"
- **Agent/NLP 教程**：使用描述性标题，如"LangGraph 底层原理与基础入门"
- **PyLearning**：格式 `Python核心语法<编号>-<主题>`，如"Python核心语法01-数据存储与运算"
- **NLP 系列**：格式 `<系列名>专题<NN>. <主题>`
- **随笔/日记**：使用简洁有力的标题，如"论持久战"

### 标签 (tags) 规则

**核心原则：优先复用已有标签。若没有合适的标签，可创建新标签，但必须经用户同意。**

1. 先读取 `references/tags-catalog.md` 查看已有标签列表
2. 根据博客内容选择 2-4 个最相关的标签
3. 算法题解**必须**包含 `Solution` 标签
4. 如果已有标签无法准确描述内容，可以提出新标签建议，但**必须在第五阶段审核时明确告知用户**，获得同意后方可使用

创建新标签时的规范：
- 中文技术标签用中文名（如 `双指针`、`哈希表`）
- 英文框架/工具名保持原样（如 `LangGraph`、`FastAPI`）
- 标签应简洁（一般 2-6 个字）
- 标签应是可复用的（不是一次性描述）

### 分类 (category) 规则

- Hot100 题解 → `category: hot100`
- Algorithm and Structure → `category: Algorithm and Structure`
- Agent 深度技术文章 → `category: Agent`
- 通用教程 → `category: Guides`
- **PyLearning / NLP / Learning / 日记 / 随笔 → 通常不加 `category`**（参考已有博客的做法）

### 描述 (description) 规则

- Hot100: `Leetcode-<题号>`（简洁）
- 普通题解: 题目来源，如 `LeetCode-2130` 或 `华为2025秋招AI岗笔试题`
- Agent/NLP: 一句话概括文章内容
- 教程: 简要描述
- 随笔/日记: `学习笔记` / `随笔` / `日记`

│   ├── 05-普通数组/
│   ├── 06-矩阵/
│   ├── 07-链表/
│   ├── 08-二叉树/
│   ├── 09-图论/
│   ├── 10-回溯/
│   ├── 11-二分查找/
│   ├── 12-栈/
│   ├── 13-堆/

**Hot100 子文件夹判断**：根据题目涉及的核心技术关键词归入。完整分类列表（17 个）：

|:--:|------|------|

| 04 | `04-子串` | 子串、子数组问题（非滑动窗口类） |
| 05 | `05-普通数组` | 一般数组操作、原地修改 |
| 06 | `06-矩阵` | 二维矩阵、旋转、螺旋 |
| 07 | `07-链表` | 链表操作、反转、环检测 |
| 08 | `08-二叉树` | 二叉树遍历、性质、构造 |
| 09 | `09-图论` | 图遍历、拓扑排序、最短路径 |
| 10 | `10-回溯` | 回溯、排列组合、子集 |
| 11 | `11-二分查找` | 二分查找、二分答案 |

- 如果题目涉及多种技术，以**主要解法**所用的技术为准
- 若题目确实无法归入现有分类，可与用户讨论新建文件夹

**Agent 子文件夹**：
- 原理/源码解析 → `原理/`
- 应用/实战 → `应用/`

### 文件命名

每个博客文章放在独立文件夹中，以 `index.md` 命名：
```
posts/hot100/01-哈希/001.两数之和/index.md
```

**命名编号规则**：
- Hot100: 全局递增编号 `NNN.标题`
- Algorithm and Structure: 全局递增编号 `NNN.标题`
- 其他类型: 可用 `NNN.标题` 或直接用 `标题`

**重要：所有文件夹/文件位置和命名在创建前必须经用户确认。**

## 第四阶段：编写博客内容

### 全局风格要求

1. **语言**：正文全部使用简体中文。代码中的变量名、函数名使用英文，注释使用中文
2. **技术术语**：英文术语保留原样（如 "Transformer"、"Token"），但首次出现时用中文解释
3. **代码块**：使用标准 markdown 围栏代码块 ```` ```python ````，代码使用 4 空格缩进
4. **数学公式**：行内用 `$...$`，块级用 `$$...$$`
5. **语气**：教学性、系统性。用"我们"而非"你"，营造共同探索的感觉
6. **作者**：始终为 `Hygen`

### LeetCode 题解专项要求（最重要）

这部分是 blog-generator 的核心能力。写 LeetCode 题解时**必须**做到：

#### a. 逐步引导的解题思路

不要直接给出最优解！按照这个思维链条展开：

1. **先给出暴力/朴素解法**：用最简单直观的方式解决，写出代码
2. **分析暴力解的问题**：计算复杂度，指出瓶颈在哪里
3. **提出优化方向**：解释"我们可以怎样改进"，引出优化思路
4. **逐步实现优化解**：每一步解释"为什么这样做"，用代码片段辅助
5. **给出最终完整代码**
6. **复杂度分析**：时间 + 空间

这就是"降维思维"——从暴力的 $O(n^3)$ 逐步降到 $O(n^2)$ 再降到 $O(n)$（如果能）。

#### b. 执行过程可视化

对于复杂算法（双指针滑动、DP 状态转移等），用表格展示执行过程：

```markdown
| `i` | `j` | `cnt` | 操作 |
| --- | --- | --- | --- |
```

让读者能"看到"算法如何一步步工作。

#### c. 主动寻找更优算法

写完主解法后，**必须**做以下检查：

1. 是否存在时间复杂度更低的算法？（如 $O(n^2) \to O(n \log n) \to O(n)$）
2. 是否存在空间复杂度更优的算法？（如 $O(n) \to O(1)$）
3. 是否存在代码更优雅的写法？（更 Pythonic、更简洁）
4. 是否存在其他算法路线？（哈希表 vs 双指针 vs DP）

若有，添加 `# 解题思路（<替代方法名>）` 章节和 `# 代码优化` 章节，并用对比表格列出不同方法的优劣。

如果当前解已是最优（需论证为什么，如"输出本身需要 $O(n^2)$ 时间"），则在代码优化章节聚焦于常数优化和可读性。

#### d. Hot100 vs 普通 Solution 的区别

- **Hot100**：`category: hot100`，文件夹 `posts/hot100/{主题}/`，description 为 `Leetcode-<题号>`
- **普通 Solution**：`category: Algorithm and Structure`，文件夹 `posts/Algorithm and Structure/Solution/`，description 可以更灵活

通过 LeetCode URL 判断：URL 含 `top-100-liked` 即为 Hot100。

**重要：如果用户只给了 LeetCode 链接但没有提供代码**，必须提醒用户提供代码后再继续。不要自行编写题解代码——代码应由用户提供，skill 负责组织解题思路和博客结构。

### Agent / RAG 等技术教程专项要求

1. 名词首次出现必须用中文解释
2. 代码示例尽量完整可运行
3. 建议展示"实验结果"小节，贴出运行输出
4. 图构建/流程设计逐步展示，不要一次性给完整代码
5. 图片存放于 `assets/` 或 `imgs/` 子文件夹中，引用格式 `![alt](./assets/filename.png)`
6. 如果用户提供了资料（文档链接、代码仓库等），仔细阅读后再撰写

### 读后感 / 日记 / 随笔专项要求

- **读后感**：引用原文关键段落 + 个人感悟和分析。建议先概述全书/全文，再展开具体观点
- **日记**：日期 + 记录事项，自由格式
- **随笔**：自由表达，不需要结构化模板
- 这类内容**不需要**技术标签，通常**不需要** `category`

### PyLearning 教程专项要求

- 教科书式结构：定义 → 表格对比 → 代码示例 → 案例实操
- 使用数字编号层级（`1.1.1. 定义`）
- 每个知识点配完整代码示例和预期输出
- "案例实操"小节从实际场景出发设计练习

## 第五阶段：用户审核

博客写完后，执行以下步骤：

1. **展示完整博客内容**给用户阅读
2. **明确告知**以下信息：
   - 选择的博客类型
   - 存放的文件夹路径
   - 使用的 tags 和 category（如有新建标签，特别说明）
   - 文件将创建在什么位置
3. **等待用户确认**。用户可能需要修改标题、标签、内容细节等
4. 用户确认无误后，将博客写入文件

## 第六阶段：Git 同步

博客写入文件且用户完全同意后：

```bash
git add <文件路径>
git commit -m "post: <commit 信息>"
git push
```

**Commit 信息格式**（参考已有 git 记录）：
- Hot100: `new blog hot100-<编号>`
- Algorithm and Structure: `new blog AnS-<编号>` 或 `new blog Algorithm and Structure-<编号>`
- 其他: `new blog <简短描述>`

推送前告知用户将要执行的操作，获得同意后再 push。

## 图片与附件处理

博客文章可能包含图片、代码附件、数据文件等附加内容。**不要**将这些文件直接散落在 `index.md` 同目录下，而应按类型归入子文件夹：

### 附件目录结构

```
posts/{分类}/{文章标题}/
├── index.md              ← 博客正文
├── assets/               ← 截图、架构图、流程图等配图
│   ├── architecture.png
│   └── screenshot.png
├── resources/            ← 代码文件、数据文件、配置文件等
│   ├── example.py
│   └── data.json
└── imgs/                 ← 备选：专门放图片（当图片较多时，可以从 assets 中分离）
    └── diagram.png
```

### 使用规则

1. **少量图片（≤3 张）** → 放在 `assets/` 文件夹
2. **图片较多（>3 张）** → 可单独建立 `imgs/` 文件夹，`assets/` 放其他类型附件
3. **代码附件/数据文件** → 放在 `resources/` 文件夹
4. **图片引用格式**：`![描述](./assets/文件名.png)` 或 `![描述](./imgs/文件名.png)`
5. **远程图片**：仍使用已有 OSS URL（如 `https://skojiangdoc.oss-cn-beijing.aliyuncs.com/...`），无需下载到本地
6. 不要在博客中引用不存在的图片

### 命名规范

- 图片和附件使用有意义的英文名或拼音名，避免使用 UUID 和中文文件名
- 示例：`architecture.png`、`screenshot-result.png`、`example-solution.py`

## 参考现有博客风格

在编写博客前，**必须先读取至少一篇同类型的已有博客**，确保风格一致：

- 写 Hot100 博客前 → 读 `posts/hot100/` 下任意一篇 `index.md`
- 写 Agent 博客前 → 读 `posts/Agent/LangGraph/` 下任意一篇
- 写 PyLearning 博客前 → 读 `posts/PyLearning/` 下任意一篇
- 等等

这个步骤很关键——它让你校准语气、代码风格、结构细节，而不是凭空猜测。
